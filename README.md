# ЦПИ CRM — локальный MVP

Веб-сервис единой базы участников ЦПИ по требованиям
[`CODEX_CRM_CPI_MVP.md`](./CODEX_CRM_CPI_MVP.md). Это модульный TypeScript-монолит:
Next.js-интерфейс, Fastify API, PostgreSQL, worker, Keycloak, Redis и приватное
S3-совместимое хранилище MinIO.

## Что уже работает

- OIDC-вход и централизованная матрица ролей/permissions;
- серверный реестр участников с поиском, фильтрами и cursor pagination;
- реестр мероприятий: в мероприятии видны его участники, комментарии и материалы;
- карточка участника с контактами, принадлежностью, всеми мероприятиями, комментариями,
  артефактами и полным разрешённым набором полей исходной книги;
- артефакты с привязкой к участнику и мероприятию, неизменяемыми версиями, явно
  выбранными авторами и текстом/ссылками/файлами;
- append-only оценки конкретной версии целым числом `1..10`;
- единый доменный расчёт активации и активности на точных границах 252/504 часа;
- приватные upload intent, quarantine, SHA-256, MIME и ClamAV-проверка файлов;
- дашборд без финансовых метрик;
- консервативное автоматическое разрешение дублей с журналом решений и обратимым merge;
- очистка очевидных тестовых, служебных и бессодержательных ФИО без потери import audit;
- CSV-экспорт всех участников, участников мероприятия и текущей отфильтрованной выборки;
- immutable import staging с редактированием чувствительных значений, field-level provenance и идемпотентный импорт;
- PostgreSQL outbox, worker, фоновые переходы lifecycle и ночная сверка;
- аудит мутаций, OpenAPI, health/readiness.

Основной интерфейс намеренно построен вокруг двух сущностей: «Участники» и
«Мероприятия». Отдельные вкладки «Задачи», «Артефакты» и «Дубли» убраны. Артефакты
доступны в контексте участника и мероприятия; внутренние таблицы операций,
версий, задач и merge-history сохранены для целостности, аудита и отката.

## Результат проверки исходной книги

Исходник: `Участники_всех_мероприятий_Стартап_студии_ЯДРО1.xlsx`.

| Контроль                               |         Результат |
| -------------------------------------- | ----------------: |
| Листы                                  |           34 / 34 |
| Неизменяемые source rows               |            11 739 |
| Извлечённые raw cells                  |           208 330 |
| Person observations                    |            12 122 |
| Принятые / отклонённые observations    |       12 110 / 12 |
| Игнорируемые служебные source rows     |                11 |
| Чувствительные ячейки, отредактировано |             1 991 |
| Формулы, сохранённые только как текст  |                46 |
| SHA-256                                | `340ed6a5…e9c179` |

На чистой PostgreSQL первый commit создаёт 2 898 валидных исходных профилей,
5 606 контактов, 68 мероприятий и 12 021 факт участия. Также импортируются 56
артефактов, 56 их неизменяемых версий и 118 связей с авторами. Для сверки остаются
60 090 provenance-связей и 36 140 field observations.

После импорта политика `AUTO_DEDUPE_V1` рассматривает 1 602 кандидатные связи:
1 252 получает решение `MERGED`, 5 — `NOT_DUPLICATE`, 345 — `DISMISSED`. Всего 785
неканонических профилей присоединяются к детерминированно выбранным каноническим
карточкам; в рабочем реестре остаётся 2 113 участников и 0 открытых кандидатов.
Merge хранит состав операции и может быть отменён. Повторный commit той же книги не
создаёт ни новых исходных записей, ни доменных сущностей, ни решений дедупликации.

Все 11 739 исходных строк остаются в immutable staging, включая 11 строк, которые
не создают рабочие сущности из-за очевидно тестового, служебного или бессодержательного
ФИО. СНИЛС (1 930 значений) и даты рождения (61) редактируются до записи raw staging:
в БД остаются координаты ячейки, заголовок и hash исходной строки, но не само значение.
Они не выводятся в карточку и не индексируются. Все остальные поля доступны
пользователю с соответствующим permission. Формулы Excel не исполняются: 46 формул
сохраняются как текст, без доверия к cached result.

Экспорт отдаёт UTF-8 CSV и повторяет серверные фильтры реестра: можно выгрузить всех
канонических участников, только участников выбранного мероприятия либо текущую
выборку поиска/фильтров. Выгрузка защищена permission `exports.bulk`, записывается
в аудит, формируется потоково, имеет `Cache-Control: no-store` и не раскрывает
отредактированные чувствительные значения.

## Быстрый локальный запуск

Нужны Node.js 22+ (для пилотного сервера рекомендуется Node.js 24), Corepack и Docker
Compose. Текущий пользователь должен иметь доступ к Docker socket.

```bash
corepack enable
corepack pnpm local:setup
corepack pnpm dev
```

`local:setup` поднимает зависимости, применяет forward-only миграцию, создаёт
локальные настройки и идемпотентно вносит всю исходную книгу. Затем открыть:

- CRM: <http://localhost:3000>
- OpenAPI: <http://localhost:3001/documentation>
- Keycloak: <http://localhost:8080>
- MinIO console: <http://localhost:9001>

Для входа в CRM нажать «Войти через ЦПИ ID» и использовать realm-пользователей:

- `admin / admin` — полный доступ;
- `manager / manager` — операционная работа.

Это не учётная запись админки Keycloak. Для Administration Console на
`localhost:8080` используются `local-admin / local-admin-change-me`. Для
MinIO console — `cpi-minio / cpi-minio-local-secret`.

Пароли и секреты из dev-контура нельзя переносить на сервер. Для локального запуска
они находятся в `infra/docker-compose.yml` и Keycloak realm только ради
воспроизводимости.

Остановить инфраструктуру:

```bash
corepack pnpm infra:down
```

Команда останавливает контейнеры, но не удаляет named volumes: PostgreSQL,
MinIO, Redis, база Keycloak и сигнатуры ClamAV сохраняются между
запусками. Realm JSON импортируется при первом создании realm и не
перезаписывает его при обычном повторном запуске.

## Ручной запуск по шагам

```bash
corepack pnpm install
corepack pnpm infra:up
corepack pnpm build
corepack pnpm db:migrate
corepack pnpm db:seed
corepack pnpm import:workbook -- --dry-run --report-dir ./import-reports/dry-run
corepack pnpm dev
```

Для commit через CLI нужны UUID организации и системного импортёра. Однокомандный
`local:setup` получает их из БД автоматически. В интерфейсе data steward может
выполнить тот же поток `dry-run → commit` на странице «Импорт».

## Проверки

```bash
corepack pnpm format:check
corepack pnpm typecheck
corepack pnpm test
corepack pnpm build
```

Тесты импортёра читают реальную книгу, но не выводят персональные данные. Для
интеграционной проверки миграции нужен PostgreSQL 18 либо контейнер из Compose.

## Настройка окружения

Основные переменные перечислены в [`.env.example`](./.env.example). Приложение имеет
безопасные local-defaults. Для ручной настройки скопировать шаблон в корне:

```bash
cp .env.example .env
```

Корневой `.env` читают API, worker, Next proxy, Drizzle migration и CLI импорта.
Относительный `IMPORT_WORKBOOK` разрешается от корня репозитория. `local:setup`
всегда готовит именно штатную локальную PostgreSQL из Compose на порту `5433`;
внешние endpoint’ы настраиваются в ручном/серверном сценарии.

Ключевые переменные:

- `DATABASE_URL`, `REDIS_URL`;
- `OIDC_ISSUER`, `OIDC_CLIENT_ID`, `OIDC_CLIENT_SECRET`, `OIDC_REDIRECT_URI`;
- `SESSION_KEY_BASE64` — ровно 32 случайных байта в base64;
- `S3_*`, `CLAMAV_*`;
- `ARTIFACT_BASELINE_AT`, `ORGANIZATION_TIMEZONE`;
- `IMPORT_WORKBOOK`.

## Перенос на сервер

Код не привязан к локальным контейнерам. Для пилота заменяются endpoint’ы и секреты:

- PostgreSQL → управляемый PostgreSQL либо отдельный сервер;
- MinIO → приватный S3-compatible bucket;
- Keycloak → корпоративный OIDC;
- reverse proxy → Caddy/Nginx с TLS и одним origin для web/API.

До пилота обязательны: утверждение `artifact_baseline_at`, production claim mapping,
смена всех секретов, ежедневный зашифрованный backup PostgreSQL/объектов/IdP и
проверка восстановления. Цели из ТЗ: RPO ≤ 24 часа, RTO ≤ 8 часов.

## Ограничения текущей поставки

- Production не разворачивался и внешняя инфраструктура не изменялась.
- Полный browser E2E с настоящими Keycloak, MinIO и ClamAV требует доступного Docker
  daemon. В текущем окружении Docker socket недоступен; unit-тесты, сборка, миграции,
  API vertical flow и двукратный импорт на PostgreSQL проверены отдельно.
- Финансы, сделки, боты, ML и развитая коммерческая CRM намеренно не входят в MVP.
